01

Posicionamento do site

Categoria

AI coding workflow observability — uma nova categoria, não "dev tool", não "token saver", não "analytics".

Percepção pretendida

Orbit é a camada de visibilidade operacional que falta entre "o dev usa IA" e "o time entende como a IA está sendo usada". É o equivalente a um APM, mas para sessões de IA em vez de serviços web.

Dores que resolve

  • Uso de IA vira caixa-preta: ninguém sabe se está funcionando ou desperdiçando.
  • Decisões sobre ferramentas de IA feitas por feeling, sem evidência.
  • Rework invisível: o dev não percebe que está iterando 5 vezes onde bastava 1.
  • Custo cresce sem explicação mecânica e sem correlação com resultado.
  • Times não conseguem distinguir uso real, uso passivo e desperdício estrutural.

Resultados que entrega

  • Visibilidade contínua do que acontece dentro de sessões de AI coding.
  • Diagnóstico em tempo real com causa mecânica, não opinião.
  • Evidência auditável de cada decisão (append-only log).
  • Padrões de desperdício classificados por risco.
  • Comparativo antes/depois com métricas reais, não estimadas.

Linguagem

Direta, técnica, mensurável. Frases curtas. Substantivos concretos. Verbos operacionais (detect, record, diagnose, reconcile, prove). Zero hype.

Erros de posicionamento a evitar

  • Vender como "economize tokens" (commoditiza e vira race-to-the-bottom).
  • Vender como dashboard (põe Orbit no balde de BI).
  • Vender como linter ou code quality (escopo errado).
  • Vender como "AI para gerenciar sua AI" (vaporware).
  • Vender como ferramenta de produtividade individual (limita ticket).

03

Objetivo da home

Fluxo mental em 5 batidas:

  1. Captar atenção (0–3s): hero mostra o que é, em uma frase concreta, com prova visual ao lado.
  2. Explicar (3–15s): "how it works" em três passos mecânicos, não vagos.
  3. Provar (15–45s): antes/depois numérico + produto em ação com screenshot real.
  4. Convencer (45–90s): casos de uso por papel + seção de confiança/rastreabilidade.
  5. Converter (90s+): CTA final, FAQ para quebrar objeções, footer com caminhos de compra/install.
Golden rule
Cada seção deve responder à dúvida que a anterior provocou. Se uma seção não quebra objeção nem gera evidência, ela sai.

5.3 The problem

Objetivo: fazer o visitante se reconhecer no problema.

Título: You can't improve what you can't see.

Subtítulo: Most AI coding sessions are invisible. The waste is mechanical — and fixable — if you can observe it.

Conteúdo: 3 colunas:

  • Invisible rework — You don't notice you corrected the AI four times on the same file.
  • Opaque cost — Token bills grow without explanation. Nobody knows which session burned the budget.
  • Anecdotal decisions — "AI feels slower this week" is not a signal. It's a feeling.

5.4 How it works

Título: How Orbit works.  Subtítulo: Three steps. No dashboards to configure. No integrations to build.

  1. Install once. Drop the skill into your AI assistant. 30 seconds.
  2. Work normally. Orbit reads the session as it happens. It never modifies your code.
  3. Get diagnoses on demand. When waste patterns appear, Orbit tells you what, how risky, and what to do next.

Provas e credibilidade

Sem logos famosos, construa confiança com sinais técnicos, não sociais.

Sinais obrigatórios no site

  • Threat model público em /security/threat-model.
  • Schema JSON versionado e documentado (orbit.doctor.v1, orbit.session.v1).
  • Changelog público com datas reais, não releases vagas.
  • Evidence methodology explicando como before/after foi medido.
  • Self-hosting guide (mesmo que experimental).
  • Status page mínimo (pode ser estático no lançamento).

Mensagens de credibilidade

Built in the open. Every decision is logged, versioned, and auditable.

No prompts stored. No code uploaded. Ever.

Schema is versioned. Integrations don't break on upgrade.

Threat model and privacy assumptions are public.

Evidência de maturidade operacional

  • CLI formal com subcomandos nomeados (run, stats, doctor, analyze, ui, quickstart).
  • Flag --json em comandos críticos (atomic write).
  • HMAC opcional via ORBIT_HMAC_SECRET.
  • Teste automatizado de UX (ux_audit_test.go).
  • Governança de schema (SCHEMA-EVOLUTION.md).

Como evitar parecer vaporware

  • Mostrar output real em todas as capturas.
  • Citar nomes de comandos e flags que existem de verdade.
  • Linkar código-fonte público se aplicável.
  • Usar datas absolutas no changelog (Apr 17, 2026), não relativas.
09

FAQ oficial

Regra: cada resposta abre com uma frase completa e autossuficiente.

1. What is Orbit?

Orbit is an operational visibility layer for AI coding sessions. It watches how you work with tools like Claude Code, GPT, and Gemini, detects waste patterns in real time, and records every decision in an append-only evidence log.

2. Who is Orbit for?

Developers, tech leads, and engineering teams who use AI coding assistants daily and want to understand — with evidence — how those sessions actually run. If AI is an occasional tool for you, Orbit is probably overkill.

3. Is Orbit tied to Claude Code?

No. Orbit works with Claude Code, GPT, Gemini, and any AI coding assistant that exposes a session stream. Claude Code is the first-class integration because of its skill system, but the detection engine is assistant-agnostic.

4. Does Orbit store my prompts?

No. Orbit analyzes patterns in the session, not the text of your prompts. Nothing you type is stored, exported, or sent to a third party. Default deployment is entirely local.

5. Does Orbit store my code?

No. Orbit never reads file contents beyond what's already in the session. No repository is cloned, no file is uploaded. References are kept as paths, not bodies.

6. What's the difference between "cost" and "value" in Orbit's view?

Cost is what you pay in tokens. Value is what the session produced. Orbit makes both observable — and shows when the first grows without the second. It doesn't just measure spend, it measures efficiency.

7. How is this different from a token dashboard?

A token dashboard tells you how much you spent. Orbit tells you why — which behaviors in the session caused the cost, and which specific action would have prevented it. It's diagnosis, not accounting.

8. Can I use Orbit as an individual?

Yes. The free tier is designed for individual use: full pattern detection, real-time diagnoses, no account, no login. You install it once and it works.

9. Can a team use Orbit?

Yes. Orbit Pro aggregates patterns across a team's sessions without exposing prompts or code. Leads see patterns and risks at a team level; developers keep full privacy inside their sessions.

10. How do I install it?

Drag the skill/ folder (or SKILL.md) into your AI assistant. That takes around 30 seconds. The first diagnosis appears on your next complex session. Full guide in /docs/quickstart.

11. Does Orbit integrate with my existing tools?

Yes. Orbit emits versioned JSON that any pipeline can consume. The CLI is designed to pipe into your own dashboards, OpenTelemetry collectors, or internal ingestion systems.

12. Can I self-host Orbit?

Yes. The engine is a standalone service (HTTP on localhost by default). Docker Compose and deployment notes are in /docs/self-hosting. Your data never leaves your infrastructure.

13. How long before I see value?

One session. The first time Orbit detects a waste pattern during your session, you'll see a diagnosis with the action to take. Most users see that in under 15 minutes of normal use.

14. Is Orbit secure?

Yes. Orbit runs locally by default, supports HMAC-signed events on the tracking channel, and exposes an append-only evidence log that cannot be rewritten silently. The public threat model explains every assumption.

15. What does Orbit actually record?

Orbit records: detection events, risk scores, recommended actions, reconciliation entries, and schema-versioned diagnoses. It does not record: your prompts, your code, your file contents, or any personally identifiable session content.

16. What makes Orbit different?

Three things. It diagnoses, it doesn't just count. It records evidence, it doesn't just report. And it treats an AI session as something you can observe, not something you have to guess about after the fact.

Tipografia

  • Sans: Inter (ou Geist Sans) — pesos 400, 500, 600.
  • Mono: JetBrains Mono ou Berkeley Mono — 400 e 600.
  • Serif: nenhuma.
  • Escala: 12, 13, 14, 16, 18, 20, 24, 32, 40, 56, 64. Nunca tamanhos fora dessa escala.

Densidade e espaçamento

  • Grid: 12 colunas, 1200px max, gutter 24px.
  • Espaçamento vertical entre seções: 96px (desktop), 64px (tablet), 48px (mobile).
  • Cards: radius 12px, padding 24px, border 1px.
  • Botões: radius 8px, padding 12×20, altura 44px.

Estilo de ilustrações / mockups

  • Prioridade absoluta: terminal mockups com output real.
  • Segundo: diagramas minimalistas em linha fina (1.5px), 3–4 cores máximo.
  • Proibido: ilustrações 3D genéricas, personagens estilizados, blobs coloridos.

Ícones, animações, responsividade

  • Ícones: linha fina (1.5px stroke), 24px padrão, uma família única (Lucide ou Phosphor).
  • Animações: 180–250ms, easing cubic-bezier(0.22, 1, 0.36, 1). Proibido scroll-jacking, parallax pesado, autoplay de vídeo.
  • Breakpoints: sm: 640, md: 768, lg: 1024, xl: 1280.
  • Home deve converter em 375px. Terminal vira carrossel em mobile. CTA primário sticky no fundo.

Tom de voz

Como Orbit fala

Como um engenheiro sênior que respeita o tempo do leitor. Direto, específico, sem drama. Usa números quando tem; não usa quando não tem.

Evitar

  • Palavras ocas: "leverage", "unlock", "empower", "revolutionary".
  • Hype de IA: "AI-powered", "intelligent", "next-gen", "smart".
  • Promessas vagas: "save time", "boost productivity".
  • Superlativos sem prova.
  • Jargão acadêmico desnecessário.

Reforçar

  • Verbos mecânicos: detect, record, diagnose, reconcile, observe, prove.
  • Substantivos concretos: session, pattern, evidence, action, risk.
  • Números com unidade explícita e fonte linkável.
  • Reconhecimento honesto do escopo ("Orbit doesn't review your code — it reviews the session").

Humano e técnico simultaneamente

  • Frases curtas alternadas com uma frase média. Nunca três longas em sequência.
  • Use "you", não "the user".
  • Use contrações ("it's", "doesn't").
  • Abra seções com o ponto principal; não atrase o valor com setup.

Confiante sem arrogante

  • Reivindicar apenas o que o produto faz.
  • Mostrar limites: "Skip Orbit if…".
  • Deixar o leitor decidir — sem escassez falsa ("last spots!").

Veredito final

O que torna esse site forte

  • Uma categoria nova nomeada ("observability for AI coding sessions") em vez de briga diluída com token dashboards.
  • Evidência visível em cada seção: output real de comandos reais, evidence ids, schema versionado — o visitante sente que o produto existe.
  • Copy específica, com verbos mecânicos e substantivos concretos, livre de hype.
  • Honestidade de escopo (Is / Skip) que paradoxalmente aumenta conversão.
  • Privacidade como arquitetura, não como selo.
  • Fluxo de conversão limpo: um CTA primário, repetido, consistente.

O que pode destruir a conversão

  • Tratar "token economy" como manchete → commoditiza o produto.
  • Encher a home de ilustrações genéricas de startup SaaS → mata a credibilidade técnica em 2 segundos.
  • Dashboards fake ou números inflados sem link para metodologia → vira vaporware percebido.
  • FAQ evasiva sobre privacidade ("we value your data") → comprador técnico desiste imediatamente.
  • Hero com duas ou três propostas competindo → dilui o 10-segundos-test.
  • Mostrar roadmap ou features que não existem → perde o "parece pronto".

Prioridade absoluta da V1

Fazer a home executar perfeitamente três coisas, nessa ordem, com zero compromisso:

  1. Explicar em 10 segundos o que é Orbit — "operational visibility for AI coding sessions", com terminal real ao lado.
  2. Provar em 45 segundos que o produto existe e mede de verdade — orbit run + tabela before/after + evidence ids.
  3. Converter em um clique — Install the free skill, sem conta, sem formulário, sem atrito.

Se a V1 acertar essas três, tudo depois (teams, integrations, case studies) é extensão natural. Se errar, nenhum recurso posterior compensa.
